# Backlog plugin — design spec

**Дата:** 2026-06-13
**Статус:** утверждён к реализации

## 1. Проблема

У Claude Code нет единого, надёжно срабатывающего рефлекса для отложенных задач. Деферралы складываются хаотично — то в память, то в случайные файлы, то теряются. Нужен **forcing function**: один маршрут, один формат, срабатывающий в момент «отложить».

Ценность плагина — в **надёжности рефлекса capture** и едином известном месте хранения, а не в самом хранилище.

## 2. Цели и не-цели

**Цели:**

- Единый автоматический маршрут записи отложенных задач (capture-рефлекс).
- Хранение в репозитории проекта, человекочитаемое, версионируемое.
- Просмотр/сортировка бэклога и управление статусом.
- Чёткое разграничение «бэклог» vs «следующий этап открытого плана».

**Не-цели (сознательно вне scope):**

- **Исполнение задач.** Плагин не исполняет пункты бэклога. Когда пользователь берёт пункт в работу, исполнение идёт обычным workflow пользователя (триаж → debugging/brainstorming/TDD). Отдельный «движок исполнения» создавал бы параллельный маршрут, конфликтующий с этим workflow.
- Батч-прогон, авто-закрытие задач, БД, внешние сервисы (GitHub Issues), индекс-кеши.

## 3. Решения по требованиям (зафиксированы с пользователем)

| Тема                | Решение                                                                              |
| ------------------- | ------------------------------------------------------------------------------------ |
| Хранилище           | Файлы в `docs/backlogs/` репозитория (не Issues, не БД)                              |
| Установка           | Глобально, чтобы рефлекс работал во всех проектах                                    |
| Capture-поведение   | **A** — авто-запись с уведомлением, **только явные деферралы**                       |
| Исполнение          | Нет в плагине; решает пользователь своим workflow                                    |
| Важность            | `low` / `medium` / `high`                                                            |
| Статус              | `open` / `done` / `cancelled`                                                        |
| Сортировка/фильтр   | В CLI-скрипте; по умолчанию только `open`                                            |
| Надёжность рефлекса | **E2** — + минимальный SessionStart-хук                                              |
| Граница план/бэклог | Критерий — принадлежность scope активного плана; узкий переспрос при двусмысленности |

## 4. Архитектура — три слоя

### 4.1 CLI-движок — `scripts/backlog.py`

Python 3, **только stdlib, без зависимостей**. Единственное место, знающее формат. Детерминированно создаёт/читает/сортирует/обновляет записи. Тайм-штампы (`created`/`updated`) проставляет сам — обе даты ведутся автоматически.

**Расположение целевой папки:** `<корень git-репо или cwd>/docs/backlogs/`. Создаётся при отсутствии. Резолв: подняться до git-root (`.git`), иначе cwd.

**Подкоманды:**

- `add --title T --priority P [--body B]` — создаёт новый файл. Генерирует `id` = (max существующий id) + 1, скан папки; пустая папка → id `1`; «дырки» в нумерации не переиспользуются (max+1, а не первый свободный). `status=open`, `created=updated=now`. Если `--body` не передан — читает из stdin. Печатает id созданной записи.
- `list [--status open|done|cancelled|all] [--sort priority|date] [--priority P]` — по умолчанию `--status open`, `--sort priority`. Печатает выровненную таблицу в stdout (колонки: `#`, `PRI`, `STATUS`, `AGE`, `TITLE`). `AGE` от `created` (`2d`, `11d`, `3h`). Сортировка `priority`: high→medium→low, внутри уровня — старые выше (по `created` asc). Пустой результат → дружелюбная строка «Бэклог пуст / нет open-задач».
- `show <id>` — печатает полную запись (frontmatter + body) для взятия в работу.
- `done <id>` — `status=done`, бамп `updated`.
- `cancel <id>` — `status=cancelled`, бамп `updated`.
- `update <id> [--title T] [--priority P] [--body B]` — правит указанные поля, бамп `updated`.

**Обработка ошибок:** несуществующий id → ненулевой exit + понятное сообщение в stderr. Битый/непарсимый файл в папке → `list` его пропускает с warning в stderr, не падает. Неизвестный priority/status во входе → отвергается с ошибкой.

### 4.2 Формат файла

Один файл на задачу: `docs/backlogs/<id>-<slug>.md`. `slug` — из title (kebab, латиница/цифры, обрезка). Пользовательский хендл — целочисленный `id`.

Frontmatter — **плоский `key: value`** (не вложенный YAML), парсится stdlib без зависимостей:

```markdown
---
id: 5
title: Вынести валидацию формы в useFormValidation
priority: medium
status: open
created: 2026-06-13T12:30:00Z
updated: 2026-06-13T12:30:00Z
---

Сейчас валидация размазана по SignupForm.tsx:40-120. Нужно извлечь
в хук useFormValidation. Контекст: обсуждали при работе над формой,
отложили чтобы не раздувать PR. Затрагивает только SignupForm.
```

- `created`/`updated` — UTC ISO 8601.
- Body — markdown произвольной длины; обязан содержать достаточный контекст для возврата без повторного анализа кодовой базы (что сделать, почему отложили, где — ссылки `file:line`).

### 4.3 Скилл — `skills/backlog/SKILL.md` (forcing function)

**Триггер (description) срабатывает в двух случаях:**

1. **Явный деферрал** в разговоре. Маркеры: «потом / позже / отложим / не сейчас / вне scope этого PR / TODO на будущее / сделаем в другой раз». Гипотетику и проходные реплики НЕ ловит.
2. **Обращение к бэклогу**: «что в бэклоге», «давай разберём бэклог», «положи в бэклог».

**Секция Capture:**
На явном деферрале → вызвать `backlog.py add` с ёмким `title` и `body`, содержащим достаточный контекст. Затем уведомить: «записал в бэклог #N: …». Без вопроса (поведение A).

**Секция Plan-boundary (разграничение с открытым планом):**
Критерий — принадлежность scope **активного плана**, а не «когда сделаем».

- Задача двигает цель текущего открытого плана → **не бэклог**, остаётся в трекинге плана (TodoWrite / чек-боксы plan-doc).
- Задача — отдельная история за пределами цели плана → **бэклог**.

Детекция «есть активный план» (по убыванию надёжности):

1. Активный TodoWrite/Task-список с незакрытыми пунктами.
2. Открытый plan-doc/spec (`docs/superpowers/specs/*`, `docs/plans/*`) с невыполненными шагами.
3. Явный контекст разговора об исполнении плана.

Нет ни одного сигнала → активного плана нет → деферрал идёт в бэклог.

**Узкое исключение из A:** когда есть активный план И неоднозначно, в scope задача или нет — коротко переспросить «в план или в бэклог?». Когда план есть, но задача явно вне его цели — писать в бэклог молча.

**Секция Discovery/Read:**
На обращении к бэклогу → `backlog.py list`. На взятии пункта в работу → `backlog.py show <id>`, затем **перечитать реальный код по ссылкам из записи** (защита от протухания); если устарела/уже сделана/потеряла смысл — остановиться и сказать, не исполнять слепо. Дальше — обычный workflow пользователя; батч/по-одному решает пользователь. Плагин не исполняет.

**Секция Format reference:** раскладка папки, поля, расположение.

### 4.4 Слэш-команды — `commands/*.md`

Тонкие обёртки над движком (через `!`-вызов bash, путь `${CLAUDE_PLUGIN_ROOT}/scripts/backlog.py`):

| Команда                | Действие                                                 |
| ---------------------- | -------------------------------------------------------- |
| `/backlog:list`        | `backlog.py list`; проброс `--all`/`--sort`/`--priority` |
| `/backlog:add`         | ручное добавление по явной просьбе                       |
| `/backlog:done <id>`   | `backlog.py done <id>`                                   |
| `/backlog:cancel <id>` | `backlog.py cancel <id>`                                 |

`show`/`update` отдельных слэшей не получают — их дёргает скилл напрямую (YAGNI).

### 4.5 SessionStart-хук (E2)

Минимальный хук, инъектирующий в начало каждой сессии одну строку-напоминание: «Отложил задачу за пределы текущего плана → запиши через скилл backlog (`docs/backlogs/`)». Делает рефлекс всегда-живым. Цель — надёжность, ровно против исходной боли непоследовательности.

## 5. Раскладка плагина

```
backlog/
  .claude-plugin/plugin.json        # name=backlog, description, version
  skills/backlog/SKILL.md           # forcing function
  commands/
    list.md  add.md  done.md  cancel.md
  scripts/backlog.py                # CLI-движок (stdlib)
  hooks/                            # SessionStart-хук (E2) + регистрация
  tests/                            # тесты движка
  README.md
```

## 6. Тестирование

`scripts/backlog.py` — чистая backend-логика → **TDD**. Покрыть edge-кейсы:

- **empty** — `list`/`add` на пустой папке `docs/backlogs/`.
- **boundary** — генерация id: пустая папка → 1; «дырки» в нумерации → max+1; очень длинный title → корректный slug.
- **malformed** — битый frontmatter в файле: `list` пропускает с warning, не падает.
- **deleted-resource** — `done`/`show`/`update` по несуществующему id → ненулевой exit + сообщение.
- **concurrency** — два `add` подряд не дают коллизию id (в пределах последовательного вызова; для личного использования параллельные сессии — допустимый редкий риск, зафиксировать в README).
- валидация: неизвестный priority/status отвергается.
- сортировка: priority high→low, внутри — старые выше; фильтр по статусу.
- бамп `updated` при `done`/`cancel`/`update`.

## 7. Дистрибуция

Плагин ставится глобально (local marketplace / установка из директории `/Users/alex/Documents/backlog-skill`). Папка `docs/backlogs/` создаётся в каждом проекте при первой записи.

## 8. Открытые риски (приняты осознанно)

- **Протухание записей** — смягчается дисциплиной «перечитать код перед работой» в Discovery-секции и требованием достаточного контекста в body.
- **Параллельные сессии и коллизия id / merge-конфликты** — редки для личного использования; текстовый формат делает merge разрешимым; зафиксировать в README.
- **Шум в бэклоге** — смягчается «только явные деферралы» и лёгкостью `cancel`/удаления.
