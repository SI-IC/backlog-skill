# Backlog migrate — design spec

**Дата:** 2026-06-13
**Статус:** утверждён к реализации
**Базируется на:** [`2026-06-13-backlog-plugin-design.md`](2026-06-13-backlog-plugin-design.md)

## 1. Проблема

Capture-рефлекс ловит только **новые** деферралы вперёд. Существующие проекты уже
забиты отложенными задачами, размазанными по `CLAUDE.md`, докам и памяти Claude —
рефлекс их не видит. Нужна онбординг-команда, которая собирает уже накопленный
разрозненный бэклог в структурированное хранилище плагина и вычищает источники.

## 2. Цели и не-цели

**Цели:**

- Найти явно-помеченные отложенные задачи в фиксированном наборе источников.
- Перенести их в `docs/backlogs/` по правилам плагина (с provenance).
- Аккуратно вычистить из источников, не сломав их структуру/окружение.
- Сделать это автономно (Claude отвечает за результат), без обязательного ревью
  пользователем; безопасность — за счёт git-коммита, provenance и аккуратного
  удаления, а не за счёт ручной проверки.

**Не-цели:**

- Широкая детекция «по смыслу» из прозы (только явные маркеры).
- Inline-комментарии в коде (`// TODO`) — отдельный механизм, не v1.
- Глобальная пользовательская память `~/.claude/CLAUDE.md` (кросс-проектная).
- Обогащение body анализом кодовой базы (мигрированные записи могут быть тоньше —
  это легаси-импорт, дообогащаются при взятии в работу).

## 3. Решения по требованиям (зафиксированы с пользователем)

| Тема              | Решение                                                                             |
| ----------------- | ----------------------------------------------------------------------------------- |
| Природа           | AI-driven (детекция/удаление — суждение Claude) + детерминированный helper в движке |
| Реализация        | Отдельный скилл `skills/migrate/SKILL.md` + подкоманда движка `scan-targets`        |
| Out-of-repo scope | Только память ЭТОГО проекта                                                         |
| Детекция          | Консервативная — только явные маркеры                                               |
| Модель исполнения | Автономно, одной командой, без обязательного ревью                                  |
| Когда удалять     | Импорт раньше удаления; вся миграция — один git-коммит                              |
| Provenance        | Поле `source` + оригинальный текст в body (сеть отката для out-of-repo)             |
| Важность          | По сильным маркерам → high/low, иначе medium                                        |
| Нет git           | Предупредить и НЕ удалять (только импорт)                                           |

## 4. Что сканируем (детерминированно — движок)

Новая подкоманда `backlog.py scan-targets` печатает (по одному пути на строку)
список существующих файлов для сканирования. Скилл не переизобретает логику путей.

**In-repo** (относительно корня git-репо или cwd):

- `CLAUDE.md` / `AGENTS.md` / `GEMINI.md` на всех уровнях вложенности.
- Корневые: `TODO.md`, `TODOS.md`, `NOTES.md`, `BACKLOG.md`, `ROADMAP.md`, `FIXME.md`.
- `docs/**/*.md`.

**Out-of-repo** — только память этого проекта:

- Вычисляет `~/.claude/projects/<enc>/memory/`, где `<enc>` = абсолютный путь
  проекта с заменой `/` на `-` (пример: `/Users/alex/Documents/backlog-skill` →
  `-Users-alex-Documents-backlog-skill`). Включает `*.md` и `MEMORY.md`.
- Папки нет → молча пропускается (не ошибка).

**Исключения (всегда):**

- `docs/backlogs/**` (наше хранилище), `docs/superpowers/**` (спеки/планы).
- `.git/`, `node_modules/`, `dist/`, `build/`, `vendor/`, `target/`, `.next/`,
  generated/vendored.

`scan-targets` возвращает только реально существующие файлы.

## 5. Детекция (консервативная — суждение Claude)

Мигрируемый пункт — только однозначно отложенная задача:

- строки/буллеты с явными маркерами: `TODO`, `FIXME`, чекбоксы `- [ ]`,
  «потом / позже / отложить / в бэклог / не сейчас»;
- буллеты под заголовками: `TODO` / `Backlog` / `Future` / `Later` / `Roadmap` /
  «Задачи» / «Отложено».

НЕ мигрируем: прозу-размышления, выполненные `- [x]`, обычную документацию,
описания архитектуры/правил. Сомнение → не трогаем (лучше недобрать).

**Память типизирована** (`type: user|feedback|project|reference`): мигрируем
только записи, явно сформулированные как отложенная задача. `user` / `feedback` /
`reference` — это контекст, НЕ задачи, их не трогаем.

## 6. Поток (автономный)

1. Запустить `backlog.py scan-targets`, прочитать перечисленные файлы.
2. Консервативно найти пункты (§5).
3. Пустой результат → отчёт «нечего мигрировать», стоп.
4. Если **не git-репо** → предупредить, выполнить только импорт (шаг 5), удаление
   (шаг 6) пропустить, явно сказать об этом в отчёте.
5. **Импорт (раньше удаления):** на каждый пункт
   `backlog.py add --source <путь> --priority <p> --body <оригинальный текст>`.
   Перед добавлением — дедуп: если запись с тем же `source` и текстом уже есть,
   пропустить.
6. **Аккуратное удаление** (явная ответственность Claude):
   - вырезать только сам пункт (и его вложенные под-буллеты);
   - удалить осиротевший пустой заголовок, если секция опустела;
   - файл остаётся валидным (markdown/структура не ломается);
   - файл-источник, ставший пустым (только whitespace/заголовок) → удалить целиком;
   - **memory:** удалить файл-запись И её pointer-строку в `MEMORY.md`; починить
     висячие `[[links]]` на удалённую запись.
7. **Один git-коммит** на всю миграцию (импорт + удаление in-repo источников):
   `chore: migrate scattered backlog items into docs/backlogs`.
8. **Отчёт:** сколько перенесено, из каких источников, что удалено/пропущено,
   был ли git-режим или import-only.

## 7. Provenance (расширение движка)

`backlog.py add` получает опциональный `--source <путь>`. В формат добавляется
**опциональное** поле `source` (присутствует только у мигрированных записей;
рефлекс-записи его не имеют). Оригинальный текст пункта = body записи.

`FIELD_ORDER` дополняется `source` (после `updated`). `serialize_entry` /
`parse_entry` работают без изменений логики (поле плоское, как остальные).

```markdown
---
id: 12
title: Заменить self-rolled JWT на библиотеку
priority: high
status: open
created: 2026-06-13T10:00:00Z
updated: 2026-06-13T10:00:00Z
source: CLAUDE.md
---

FIXME: переписать auth на проверенную JWT-либу, текущая ручная подпись хрупкая
```

Память вне git — но удалить безвозвратно нельзя: оригинал реконструируется из
записи бэклога (`source` + body).

## 8. Важность (маппинг)

- `high`: `FIXME` / «срочно» / «critical» / «важно» / «блокер» / «urgent».
- `low`: «когда-нибудь» / «nice to have» / «потом как-нибудь» / «someday».
- `medium`: всё остальное (нет сильного сигнала — не угадываем).

## 9. Архитектура

- `skills/migrate/SKILL.md` — новый скилл. Триггер: команда `/backlog:migrate`
  или «разобрать старые бэклоги / собрать разрозненные TODO в бэклог». Содержит
  поток §6, правила детекции §5, правила аккуратного удаления §6.6.
- `commands/migrate.md` — prompt-style слэш-команда (не `!`-скрипт, т.к. поток —
  суждение), запускающая скилл.
- `scripts/backlog.py` — две правки: подкоманда `scan-targets`; флаг `--source`
  у `add` + `source` в `FIELD_ORDER`.

## 10. Тестирование

**Движок (unit, `tests/test_backlog.py`):**

- `scan-targets`: вычисление memory-пути от заданного cwd; включение только
  существующих файлов; применение исключений (`docs/backlogs/`,
  `docs/superpowers/`); отсутствие memory-папки → пропуск без ошибки.
- `add --source`: запись поля `source`; round-trip parse/serialize с `source`;
  запись без `--source` не содержит поля.
- Регрессия: существующие 47 тестов остаются зелёными.

**Поток (не unit — суждение Claude):** headless-смок на фикстуре-проекте:

- собрать temp git-репо с `CLAUDE.md` (секция `## TODO` с буллетами), `TODO.md`,
  `docs/notes.md`, и memory-файл с задачей + pointer в `MEMORY.md`;
- прогнать `/backlog:migrate` headless;
- проверить: записи в `docs/backlogs/` с `source`; источники аккуратно вычищены
  (пустые заголовки убраны, опустевший `TODO.md` удалён); `MEMORY.md` без битой
  ссылки; всё одним git-коммитом.

## 11. Открытые риски (приняты)

- **Ложная детекция** — смягчается консервативностью; git-коммит + provenance
  дают полный откат.
- **Удаление memory вне git** — provenance в записи бэклога = сеть отката.
- **Повторный запуск** — дедуп по `source`+текст перед `add`.
- **Тонкие записи** — осознанно: легаси-импорт, дообогащаются при взятии в работу.
